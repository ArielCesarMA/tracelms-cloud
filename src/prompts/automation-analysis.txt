### A. Review requirements and test scenarios

### B. Prioritize high-value test cases

### C. Exclude poor automation candidates

### D. Evaluate automation feasibility

### E. Calculate automation ROI
Consider: manual execution effort, execution frequency, maintenance effort, automation development effort, defect detection value, business risk reduction, time savings per release.

ROI formula (use as guidance):
  ROI = (Manual Execution Cost × Annual Frequency) − (Development Cost + Annual Maintenance Cost)

ROI scale: High (rapid payback) | Medium (moderate return) | Low (limited benefit) | Negative (cost exceeds benefit) | Evidence Required (data unavailable)

### F. Categorize by automation layer — apply the test pyramid; automate at the lowest viable layer

| Priority | Layer | Target Coverage |
|----------|-------|-----------------|
| 1 | Unit | 60–80% — business rules, validation, calculations, utility functions |
| 2 | API | 15–30% — service contracts, integration, auth, error handling, data validation |
| 3 | UI | 5–15% — end-to-end journeys, cross-browser, critical user workflows only |

Decision rules (must be followed):
1. Prefer Unit over API when business logic can be validated without integration dependencies.
2. Prefer API over UI when workflows can be validated through service interfaces.
3. Reserve UI only for critical end-to-end journeys and user interaction validation.
4. Avoid duplicate coverage across layers unless risk justifies redundancy.
