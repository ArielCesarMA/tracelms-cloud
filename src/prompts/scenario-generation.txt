## EXHAUSTIVE COVERAGE MANDATE

**Generate ALL scenarios derivable from the requirements.** There is no cap on output volume. The team will review and defer what they do not need — your job is to ensure nothing testable is missed.

For every feature, function, or behaviour described in the requirements, derive scenarios across every applicable type:

| Code | Type | Definition | Derive when… |
|------|------|------------|--------------|
| HP | Happy Path | Ideal flow; valid inputs, all preconditions met | Always — at least one per distinct feature or user journey |
| AF | Alternative Flow | Valid deviation; different inputs/route, still succeeds | Any alternate valid input, role, or execution path exists |
| EC | Edge Case | Boundary values, timing, concurrency, rare state | Any field has a min/max, list can be empty, concurrent users implied |
| EG | Error/Exception | Invalid input or system failure; graceful handling required | Any input field, external dependency, or system state can fail — one scenario per distinct failure mode |
| BR | Business Rule | Verify documented rule enforcement | Any documented constraint, policy, role restriction, or business logic exists |

Each distinct failure mode, boundary condition, and business rule is a scenario surface. Do not stop at one scenario per surface — multiple independent failure modes require separate scenarios.

The only legitimate reason to generate fewer scenarios is if the requirements genuinely contain no derivable path for a given type.
